01

MySQL vs Firestore

Por que a modelagem é diferente e como isso muda sua forma de pensar

A maior diferença não é técnica — é mental. No MySQL você pensa na estrutura dos dados. No Firestore você pensa nas telas do app.

🐬 MySQL — Como você pensa
  • Quais entidades existem?
  • Como evitar repetição de dados?
  • Qual é o relacionamento entre tabelas?
  • Como fazer o JOIN mais eficiente?
🔥 Firestore — Como você pensa
  • Quais telas o app vai ter?
  • Quais dados cada tela precisa de uma vez?
  • Quantas leituras essa tela vai fazer?
  • O que vale duplicar para ler mais rápido?

Exemplo: portfólio do investidor

Imagine a tela que mostra todos os tokens que o João possui, com nome e preço atual de cada startup.

MySQL — 1 query com JOIN, banco garante tudo
-- 3 tabelas separadas, dados nunca duplicados
SELECT u.name, s.name, s.current_price, h.quantity
FROM   holdings h
JOIN   users    u ON h.user_id    = u.id
JOIN   startups s ON h.startup_id = s.id
WHERE  u.id = 42;
-- resultado: tela montada com 1 query
Firestore sem denormalização — N+1 leituras
// 1ª leitura: holdings do usuário
const holdings = await db.collection('holdings')
  .where('userId', '==', 'usr_joao').get();
// retornou 3 holdings com startupId: "abc", "xyz", "def"

// 2ª, 3ª, 4ª leituras: uma por startup!
const startupA = await db.collection('startups').doc('abc').get();
const startupB = await db.collection('startups').doc('xyz').get();
const startupC = await db.collection('startups').doc('def').get();
// 4 leituras no total. Se tiver 10 startups → 11 leituras
Firestore com denormalização — 1 leitura só
// holdings já tem o nome da startup dentro
const holdings = await db.collection('holdings')
  .where('userId', '==', 'usr_joao').get();
// retornou:
// { startupName: "AgroSense", quantity: 50, currentPrice: 12.50 }
// { startupName: "EduTech",   quantity: 20, currentPrice: 8.00  }
// tela montada com 1 leitura

02

Relações N:N no NoSQL

Elas existem — mas precisam ser resolvidas na modelagem

No MySQL, uma relação N:N é resolvida automaticamente com uma tabela pivot. No Firestore, não existe suporte nativo — você resolve na modelagem com 3 estratégias diferentes.

Estratégia 1 — Array de IDs

Guarda uma lista de referências dentro do documento. Simples, mas limitado.

Firestore — array de refs dentro do documento
// startups/stk_agrosense
{
  "name": "AgroSense",
  "investorIds": ["usr_joao", "usr_maria", "usr_pedro"]
}

// ✅ fácil: "quem investe nessa startup?"
db.collection('startups').where('investorIds', 'array-contains', 'usr_joao')

// ❌ difícil: "em quais startups o João investe?" → não funciona direto
// ⚠️  limite: array com muitos IDs fica pesado

Estratégia 2 — Coleção intermediária (junction collection)

Equivalente à tabela pivot do SQL. É o padrão mais correto quando você precisa consultar pelos dois lados.

Firestore — coleção holdings como pivot entre users e startups
// holdings/usr_joao_stk_agrosense
{
  "userId":    "usr_joao",
  "startupId": "stk_agrosense",
  "quantity":  50,
  "avgCost":   11.20
}

// ✅ "quais startups o João tem tokens?"
db.collection('holdings').where('userId', '==', 'usr_joao')

// ✅ "quem investe na AgroSense?"
db.collection('holdings').where('startupId', '==', 'stk_agrosense')
✅
No MesclaInvest, a coleção holdings resolve o N:N entre usuários e startups, e a coleção transactions resolve o N:N entre ordens de compra e ordens de venda.

Estratégia 3 — Denormalização

Duplicar os dados de ambos os lados para evitar qualquer leitura extra. Detalhado no capítulo 3.

Onde há N:N no MesclaInvest?

RelaçãoSoluçãoColeção
users ↔ startupsColeção intermediáriaholdings
ordem compra ↔ ordem vendaColeção intermediáriatransactions
users ↔ perguntas ↔ startupsCampos userId + startupIdquestions
03

O que é Denormalização

Copiar dados propositalmente para ganhar velocidade de leitura

Normalização (MySQL) = nunca repetir dado. Cada informação existe em um único lugar e você une tudo com JOIN na hora da consulta.

Denormalização (Firestore) = copiar dados propositalmente dentro do documento que vai precisar deles, mesmo criando repetição.

Visualizando a diferença

Normalizado — "AgroSense" existe em apenas 1 lugar
startups/stk_agrosense → { "name": "AgroSense" }

orders/ord_001  → { "startupId": "stk_agrosense", "quantity": 10 }
orders/ord_002  → { "startupId": "stk_agrosense", "quantity": 5  }
holdings/j_stk  → { "startupId": "stk_agrosense", "quantity": 50 }
// nome da startup não aparece em nenhum outro documento
Denormalizado — "AgroSense" duplicado onde será exibido
startups/stk_agrosense → { "name": "AgroSense" }

orders/ord_001  → { "startupId": "stk_agrosense", "startupName": "AgroSense"  }
orders/ord_002  → { "startupId": "stk_agrosense", "startupName": "AgroSense"  }
holdings/j_stk  → { "startupId": "stk_agrosense", "startupName": "AgroSense"  }
// nome duplicado — parece errado, mas é proposital

Por que vale a pena?

No Firestore você paga por leitura. Cada doc.get() é cobrado. Denormalizar transforma várias leituras em uma só.

Tela do Balcão — sem e com denormalização
// ❌ SEM denormalização — 1 read por ordem + 1 read por startup
const orders = await db.collection('orders')
  .where('status', '==', 'open').get();       // read 1

for (const order of orders.docs) {
  const startup = await db.collection('startups')
    .doc(order.data().startupId).get();       // read 2, 3, 4... N
}
// 10 ordens abertas = 11 leituras


// ✅ COM denormalização — tudo em 1 read
const orders = await db.collection('orders')
  .where('status', '==', 'open').get();       // read 1

orders.docs.forEach(order => {
  const { startupName, startupLogo, quantity, pricePerToken } = order.data();
  // tudo que precisamos já está aqui dentro
});

O custo: você precisa manter os dados sincronizados

Se o nome da startup mudar, você precisa atualizar em todos os lugares onde copiou:

Node.js backend — atualizando nome da startup em cascata
async function renameStartup(startupId, newName) {

  // 1. atualiza o documento principal
  await db.collection('startups')
    .doc(startupId).update({ name: newName });

  // 2. atualiza todas as ordens que têm a cópia do nome
  const orders = await db.collection('orders')
    .where('startupId', '==', startupId).get();

  // 3. batch: atualiza tudo de uma vez (máx. 500 por batch)
  const batch = db.batch();
  orders.forEach(doc =>
    batch.update(doc.ref, { startupName: newName })
  );
  await batch.commit();
}
⚠️
Atenção: se você denormalizar um dado que muda frequentemente (como o preço do token), vai ter que rodar esse tipo de atualização em cascata a cada mudança. Em geral, não vale denormalizar dados voláteis.
04

Quando (não) denormalizar

A decisão prática aplicada ao MesclaInvest

SituaçãoDecisãoMotivo
Dado muda raramente (nome, logo)COPIECusto de atualização baixo, ganho de leitura alto
Dado muda constantemente (preço do token)NÃO COPIETeria que atualizar em cascata a cada transação
Dado aparece em toda listagemCOPIEEvita leituras extras em cada item da lista
Dado só aparece na tela de detalheNÃO COPIE1 leitura extra para detalhe é aceitável

Decisões tomadas no MesclaInvest

orders — o que copiar e o que não copiar
{
  "startupId":    "stk_agrosense",

  // ✅ COPIADOS — mudam raramente, aparecem na listagem do balcão
  "startupName":  "AgroSense",
  "startupLogo":  "https://...",

  // ❌ NÃO COPIADO — preço muda a cada transação
  // "currentPrice" não fica aqui → busca em startups/{id} quando precisar

  "pricePerToken": 12.50, // preço que o usuário PEDIU na oferta (fixo)
  "quantity":      10,
  "status":        "open"
}
holdings — portfólio do investidor
{
  "userId":     "usr_joao",
  "startupId":  "stk_agrosense",
  "quantity":   50,
  "averageCost":11.20,

  // currentPrice NÃO fica aqui
  // a tela de portfólio faz 1 leitura adicional em startups/{id}
  // para pegar o preço atual — aceitável, é só 1 doc por startup
}

Cenários Práticos

O que acontece no banco quando cada ação ocorre no app

📝 Cenário 1 — João se cadastra no app
1

Firebase Auth cria o UID

Gera usr_joao_001 — será o ID do documento no Firestore.

2

Backend cria: users/usr_joao_001

Com walletBalance: 0 e mfaEnabled: false. Pronto.

💳 Cenário 2 — João adiciona R$ 1.500,00 de saldo fictício
1

Cria: wallet_txs / wtx_cred_001

type: "credit", amount: 1500, balanceAfter: 1500

2

Atualiza: users/usr_joao_001

walletBalance: 0 → 1500

💱 Cenário 3 — João compra 10 tokens da AgroSense de Maria (R$ 125,00)
1

Cria: orders/ord_buy_001

type: "buy", quantity: 10, price: 12.50, status: "open"

2

Backend casa com orders/ord_sell_001 da Maria

Encontra oferta de venda compatível no balcão.

3

Cria: transactions/txn_001

buyerId: joao, sellerId: maria, quantity: 10, totalValue: 125

4

Atualiza ambas as ordens

quantityFilled: 0 → 10 e status: "open" → "filled" nas duas ordens.

5

Cria/atualiza holdings dos dois

João: quantity: 0 → 10. Maria: quantity: 50 → 40.

6

Cria 2 movimentações de carteira

João: debit R$125 → saldo R$1.375. Maria: credit R$125 → saldo R$625.

7

Registra snapshot em price_history

price: 12.50, volume: 10, recordedAt: agora — alimenta o gráfico.

💡
Note que uma única ação do usuário (apertar "Comprar") dispara 7 operações encadeadas no banco. Tudo isso precisa ser feito no backend com um batch para garantir que ou tudo acontece ou nada acontece — evitando estados inconsistentes.
📈 Cenário 4 — João abre o dashboard (período: último mês)
1

Lê: holdings WHERE userId == joao

1 leitura → descobre que ele tem tokens de AgroSense e EduTech.

2

Lê: price_history WHERE startupId == agrosense AND recordedAt >= 30 dias

1 leitura por startup → retorna lista ordenada por data → gráfico de linha pronto.

08

Regras de Ouro do Firestore

Os princípios que guiam toda decisão de modelagem

🥇 Modele os dados do jeito que o app vai consumir, não do jeito que faz sentido lógico.
#RegraExemplo no MesclaInvest
11 tela = 1 leitura (meta ideal)Holdings com nome da startup já dentro
2N:N → coleção intermediáriaholdings e transactions
3Dado estável → denormalizarNome e logo da startup em orders
4Dado volátil → não duplicarcurrentPrice só em startups
5ID composto = 1 doc por par únicoholdings/{userId}_{startupId}
6Operações encadeadas → usar batchCompra de tokens (7 operações)
7Banco não garante integridade → backend garanteValidar saldo antes de debitar
8Registros imutáveis não são alteradostransactions e wallet_txs

Diferença final de mentalidade

🐬 MySQL — pergunta antes
  • Que dados existem?
  • Como evitar duplicação?
  • Qual é a forma normal mais alta?
🔥 Firestore — pergunta antes
  • Quais telas o app vai ter?
  • Quantas leituras cada tela faz?
  • O que preciso duplicar para ler menos?
🎓
Para o PI3: o foco não é usar blockchain de verdade, mas sim dominar essa forma de modelagem NoSQL, construir APIs sólidas no Node.js e integrar tudo num app Flutter funcional. A modelagem certa desde o começo evita refatorações dolorosas no meio do semestre.
